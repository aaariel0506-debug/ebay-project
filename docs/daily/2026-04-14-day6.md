## 一、做了什么

### 1. 四个新 ORM 模型

| 模型 | 主键 | 关联 | 关键字段 |
|------|------|------|----------|
| `EbayListing` | ebay_item_id | → Product.sku | listing_price, quantity_available, variants(JSON), status, last_synced |
| `Inventory` | id (autoincrement) | → Product.sku | type(IN/OUT/ADJUST/RETURN), quantity, related_order, location, operator |
| `Order` | ebay_order_id | → Product.sku | sale_price, shipping_cost, ebay_fee, buyer_country, status, order_date |
| `Transaction` | id (autoincrement) | → Order, Product.sku | type(SALE/REFUND/FEE/SHIPPING), amount, currency, amount_usd, exchange_rate |

状态枚举全部使用 `str` 而非 IntEnum（保持数据库可读性）。

### 2. Pydantic 校验 (`core/utils/validators.py`)
每个模型对应一个 `Import` schema，字段校验规则：

| Schema | 校验重点 |
|--------|----------|
| `ProductImport` | cost_price > 0，currency 在允许列表，status 有效 |
| `OrderImport` | ebay_order_id 格式校验，sale_price ≥ 0，order_date 解析多种格式 |
| `EbayListingImport` | listing_price ≥ 0，status 有效 |
| `InventoryImport` | type in {in/out/adjust/return}，quantity > 0 |
| `TransactionImport` | type in {sale/refund/fee/shipping/adjustment}，currency 校验 |

`validate_batch()`：批量校验，返回 `ImportResult(success[], errors[])`，不 crash，收集所有错误。

### 3. 迁移框架 (`scripts/migrate/base_migrator.py`)
`BaseMigrator` 抽象基类，标准流程：

```
pre_check() → migrate() [逐条 transform + 保存] → post_check() → rollback（可选）
```

子类只需实现 `name`、`model`、`transform()`，框架处理批量逻辑、异常收集和日志。

### 4. Alembic 迁移
新迁移 `3ba56797d01a`：创建 ebay_listings / inventory / orders / transactions 四张表，含外键和索引。

## 二、核心设计决策

### 为什么用 `str` 而非 IntEnum 作为 enum 列？
SQLite 的 `Enum` 类型在 Pydantic 校验和 ORM 之间传递时，`StrEnum`（Python 3.11+）直接映射到底层字符串，避免 IntEnum 转 int 再转字符串的来回转换。

### 迁移框架为什么用 `abc.ABC` 而非 Protocol？
业务模型继承明确，用 `abc.ABC` 更适合约束子类必须实现的方法（`transform`）。

### 为什么不把所有校验放在一起？
按模型分 schema，职责清晰。`validate_batch` 是通用工具，不绑定具体 schema。
