### 1. EventBus 事件总线 (`core/events/`)
- **`models.py`** — `EventLog` ORM 模型 + `EventStatus` 枚举  
  字段：id / event_type / payload(JSON) / status(PENDING→DONE/FAILED/DEAD_LETTER) / retry_count / max_retries / error_message / processed_at / handler_name + TimestampMixin  
  事件类型：STOCK_IN / STOCK_OUT / STOCK_ADJUST / ORDER_CREATED / ORDER_SHIPPED / LISTING_CREATED / LISTING_UPDATED / PRICE_ALERT / PAYMENT_RECEIVED
- **`bus.py`** — EventBus 单例，完整事件生命周期管理  
  `publish()`: 先写 event_log(PENDING) → 同步分发 handler → 重查 DB 返回最新状态  
  `_dispatch()`: 调用已注册 handler；无 handler → 立即标记 DONE；失败 → status=FAILED + retry_count++  
  `retry_pending()`: 启动时自动重试 PENDING/FAILED 事件（最多 max_retries 次）  
  `mark_dead_letter()`: 重试耗尽 → 移入 dead_letter  
  后台重试线程（可选，按 interval 扫描 failed 事件）
- 9 个事件类型已定义，开箱即用

## 二、核心设计决策

### EventBus 为什么先 commit 再 dispatch？
```
publish():
  1. session.add(EventLog(status=PENDING))  → commit
  2. _dispatch() 调用 handler                 → 可能失败
  3. handler 失败 → status=FAILED + retry_count++
  4. re-fetch → 返回最新状态
```
**好处**：事件不会因 handler 崩溃而丢失，符合"写透"原则

### 为什么不直接在 publish() 的 session 里更新状态？
因为 `_dispatch()` 需要释放 `publish()` 的 session 锁，让 handler 在独立 session 里运行，避免长事务阻塞

### 审计日志的边界是什么？
**当前**：所有 `audit_log()` 调用均记录  
**建议后续**：在 Sell Inventory API / 批量操作 / 敏感配置修改处接入
