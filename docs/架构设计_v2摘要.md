1. 项目概述
本项目旨在构建一套模块化、可扩展的eBay业务管理系统，覆盖上新、库存、财务三大核心业务。采用“先搭骨架、再填充功能”的策略，解决当前“改小错大”的耦合问题。
1.1 设计原则
模块解耦：模块之间通过事件总线通信，禁止直接import跨模块调用
统一数据层：所有模块共享core/models基础数据模型
配置中心化：一个入口管理所有配置，环境变量+YAML，杜绝硬编码
安全优先：Token加密存储，审计日志，数据与代码分离
数据可靠：事件持久化，每日自动备份，关键操作可回滚

2.2 模块通信架构
所有模块通过事件总线通信，禁止直接跨模块调用。事件先写入event_log表持久化，再分发给handler处理，处理完标记done。失败的事件可重试，避免数据不一致。
示例流程：出库操作 → inventory_offline发布 STOCK_OUT 事件 → 事件写入event_log → finance模块监听并记账 → inventory_online监听并同步eBay数量 → 各handler标记done
2.3 数据库策略
开发期使用SQLite + WAL模式（单机运行，零部署成本）。通过SQLAlchemy ORM抽象，预留切换PostgreSQL的能力。数据库文件存储在项目目录之外（~/.ebay-project/data/），代码与数据完全分离。

4. 安全性设计
4.1 Token与密钥存储
eBay Refresh Token：使用macOS Keychain（通过keyring库）存储，禁止明文存储在任何文件中
API Key/Secret：通过.env文件管理，且.env必须在.gitignore中
Access Token：仅存内存，过期自动用Refresh Token刷新
4.2 Git安全
.gitignore白名单：.env, *.db, *.sqlite, token文件, backups/, 所有包含个人数据的Excel/CSV
pre-commit hook：自动检测是否有API key、token、密码等敏感信息即将被提交
分支保护：main分支禁止force push，Jarvis/外部agent只能在feature分支操作
4.3 数据安全
数据库位置：~/.ebay-project/data/，不在项目目录内，避免意外提交
每日自动备份：cron任务每日复制数据库文件到backups/，保留近7天
关键操作前快照：数据迁移、批量操作前自动创建数据库快照
财务数据隔离：如果项目为公开repo，财务模块数据和配置绝对不进 git
4.4 eBay API权限控制
OAuth scope最小化——上新只申请sell scope，拉订单只需order scope。不要一次性申请所有权限，Token泄露时影响范围可控。
4.5 审计日志
所有敏感操作（Token刷新、数据删除、财务记录修改、eBay API调用）记录到audit_log表，包含操作人、时间、操作内容、IP地址。
5. 稳定性设计
5.1 事件总线可靠性
事件持久化：事件先写入event_log表（status=pending），再分发给handler
处理完成标记：handler处理成功后更新status=done，失败则status=failed
重试机制：启动时扫描所有pending/failed事件，自动重试（最多3次）
死信队列：重试3次仍失败的事件进入dead_letter状态，人工处理
5.2 eBay API防护
请求重试：指数退避，初始1s，最大3次
限流控制：追踪每日API调用量，近限时警告
响应缓存：相同请求短时间内不重复调用，减少API压力
离线降级：eBay API不可用时，本地操作不受影响，待恢复后自动同步
5.3 数据校验
输入校验：所有外部数据（Excel导入、eBay API返回、手动录入）经过严格校验后才写入数据库
数据迁移安全：迁移脚本具备回滚能力，迁移前后做记录数、金额校验
一致性检查：定期检查线上库存与线下库存的一致性，财务流水与订单的一致性
