## 5. 关键实现决策

### 5.1 _fetch_tracking_no 方法
- 放在 `_fetch_finances_transactions` 方法之后（§3.3 要求参考现有方法位置）
- URL: `/sell/fulfillment/v1/order/{order_id}/shipping_fulfillment`
- 异常处理：`try/except` 包裹，失败返回 `None`，记 INFO 日志
- 空响应：`fulfillments` 为空或 `None` → 返回 `None`
- 多 shipment：取 `fulfillments[0].shipmentTrackingNumber`（业务上一单一包）

### 5.2 _upsert_order 中的写入
- 新增行 `tracking_no = self._fetch_tracking_no(order_id)` 在 `buyer_paid_total` 提取之后
- existing_order 路径：`if tracking_no is not None:` 才更新（resync 场景）
- new_order 路径：`tracking_no=tracking_no` 直接传入构造函数

### 5.3 mock client 扩展
- `_mock_client` 新增 `shipping_fulfillment_responses` 参数
- `fake_get` 中根据 URL path `/shipping_fulfillment` 路由到对应响应
- 从 path 中提取 `order_id` 匹配 shipping_fulfillment_responses

### 5.4 test_resync_overwrites_tracking_no
- 测试中两次 sync 使用不同 client，模拟"先采集未发货订单，后发货再 sync"的真实场景
- 第二次查询前调用 `db_session.expire_all()` 确保从 DB 读取最新值
