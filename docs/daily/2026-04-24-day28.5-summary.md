## 为什么做这件事

Day 25-27 实装的字段解析有 **4 处和真实 eBay API 不一致**：

| # | Bug 位置 | 错误假设 | 真实情况 |
|---|---------|---------|---------|
| 1 | `order_sync_service.py:218` | `fulfillmentHrefs[0].shippingCost.value` | `fulfillmentHrefs` 经常为空数组，应读 `pricingSummary.deliveryCost.value` |
| 2 | `order_sync_service.py:512` | `lineItems[].itemTxSummaries[]` | 该字段在 Fulfillment API 响应中不存在 |
| 3 | `order_sync_service.py:522` | `GET api.ebay.com/sell/finances/v1/transactions` | 应为 `apiz.ebay.com/sell/finances/v1/transaction`（单数） |
| 4 | `client.py:233` | 单一 `settings.ebay_api_url` | Finances API 域名是 `apiz.ebay.com`，Fulfillment 是 `api.ebay.com` |

**后果**: Transaction 表的 FEE / SHIPPING 记录基本是 0 或未写入。Dashboard 毛利数字基于错误基线，一直偏离真实。

---

## 修了什么

### 1. 多 Base URL 路由（client.py + settings.py）

- `settings.py` 新增 `ebay_finances_url` property → `apiz.ebay.com`（sandbox: `apiz.sandbox.ebay.com`）
- `client.py._do_request` 判断 path 前缀：`/sell/finances/` → 走 finances URL，其它 → 走原 api URL
- **调用侧代码零改动**，路由逻辑封装在 `_do_request` 内部

### 2. shipping_cost 字段来源修正

- **旧**: `fulfillmentHrefs[0].shippingCost.value`（空数组时拿不到）
- **新**: `pricingSummary.deliveryCost.value`（订单级稳定字段）

### 3. FEE 采集重写

- **旧**: 从 `itemTxSummaries`（不存在）→ 永远空
- **新**: 主路径从 Finances API `transactions[].orderLineItems[].marketplaceFees[]` 累加
- **备用**: Finances API 失败时，从 Fulfillment API `totalMarketplaceFee.value` 兜底
- 只采 `transactionType == "SALE"`，`NON_SALE_CHARGE`（广告费）跳过，留给 Day 31-B

### 4. 新增 2 个 Order 字段

| 字段 | 类型 | 来源 | 用途 |
|------|------|------|------|
| `total_due_seller_raw` | Numeric(12,2) nullable | `paymentSummary.totalDueSeller.value` | Day 32+ 精确对账 |
| `sold_via_ad_campaign` | Boolean nullable | `properties.soldViaAdCampaign` | Day 31-B AD_FEE 采集 hint |

### 5. 新增 `_extract_buyer_paid_total` 方法

- 定义了但未调用（dead code，Day 31-B 会调用）
- 逻辑: `pricingSummary.total` + Σ `ebayCollectAndRemitTaxes[].amount`
