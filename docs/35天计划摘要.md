Day 4：eBay API加固 — 重试/限流/缓存/降级
目标：让API封装具备生产级健壮性：网络抖动能自动恢复，限流不会超额，API挂了本地照常用。
环境准备
安装缓存库：pip install cachetools
D3的eBay API基础层已就绪
数据准备
查阅eBay API限流文档：不同API的daily call limit
记录各API的rate limit值到配置文件（config/ebay_rate_limits.yaml）
功能说明
指数退避重试（core/ebay_api/retry.py）：
装饰器模式：@with_retry(max_retries=3, base_delay=1.0)
仅对网络错误和5xx状态码重试，4xx不重试
每次重试延迟翻倍：1s → 2s → 4s
添加随机抖动（jitter）避免雷群效应
所有重试记录日志
限流控制（core/ebay_api/rate_limiter.py）：
追踪每个API endpoint的每日调用次数
达到80%限额时记录WARNING日志
达到95%时拒绝新请求并抛出EbayRateLimitError
每日零点自动重置计数
调用计数持久化到数据库（重启不丢失）
响应缓存（core/ebay_api/cache.py）：
对GET请求做短期缓存（TTL可配置，默认5分钟）
相同URL+参数在TTL内直接返回缓存结果
POST/PUT/DELETE不缓存
提供手动清除缓存的方法
离线降级模式：
在EbayClient中增加is_online属性
连续3次请求失败后自动切换为offline模式
offline模式下所有eBay操作排入pending队列（写入数据库）
恢复在线后自动消化pending队列

功能说明
事件总线（core/events/bus.py）：
EventBus单例类
publish(event_type, payload)：发布事件
subscribe(event_type, handler)：订阅事件
事件先写入event_log表（status=pending），再分发给handler

事件持久化（core/events/models.py）：
EventLog ORM模型：id, event_type, payload(JSON), status(pending/done/failed/dead_letter), retry_count, created_at, processed_at, error_message

日志系统（core/utils/logger.py）：
基于loguru配置
控制台输出 + 文件轮转（每天一个文件，保留30天）
