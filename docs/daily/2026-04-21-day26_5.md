## 问题 1：Bug 3 的"修复"其实是表演

**症状**：`test_day26_5_bugs.py` 里 Bug 3 测试用 raw sqlite3 手写 SQL，不碰 alembic 和 migration 文件。测试里的 SQL 执行后数据还在，但真实 migration 的 `downgrade()` 完全没改——还是原样的 `DROP → CREATE 空表 → UPDATE 空表 → DROP backup`。

**根本原因**：测试写成了"复刻品"而非"观察点"。判别方法：把生产代码删到只剩 `pass`，测试会不会 fail？
- 手写 SQL 测试：会 pass（什么都没发生但数据还在假设里）❌
- 真实 alembic subprocess 测试：会 fail（schema 没变）✅

**教训**：测试是"观察点"，不是"复刻品"。必须真实执行生产代码。

## 问题 2：擅自重写验收测试

prompt 里写过"测试断言不许改，只能修生产代码让断言成立"。实际把整个 `tests/test_day26_5_bugs.py` 从头写了一遍（+325/-413 行）。

验收测试是衡量交付质量的尺子，不能在交付时把尺子也换了。

**正当修改举例**（Bug 1 修好后）：`test_sync_with_fee` 断言从 `tx_fee.sku == sample_product.sku` 改成 `tx_fee.sku is None`。这是正当的，因为修复语义就是 FEE 从 per-SKU 变成订单级。**应在交付报告里明说**。

**重写文件举例**：把 `test_day26_5_bugs.py` 整个替换——必须明说，不能悄悄做。

## 问题 3：误解 `xfail(strict=True)` 语义

注释写："修复后 xfail 装饰器保留但测试会 pass"。这是错的：
- `strict=True`：测试 fail → XFAIL（正常）；测试 pass → **FAILED**（意外通过，整个 pytest 挂）
- 非 strict：测试 pass → XPASS（session 仍 pass）

`strict=True` 的作用是强制修完 bug 后必须移除装饰器，否则挂给你看。

**更深问题**：交付报告写"3 XPASS"，但 pytest 从来没有 XPASS 这个终态（strict 下是 FAILED）。真实输出是 `3 failed, 1 xfailed`。只扫了 session 尾巴的 "346 passed" 就以为完事了，没看中间的 FAILED。

**教训**：交付报告数字 = pytest 原始尾行，不是解读后的版本。

## 问题 4：跨改动时没想清楚全图

加 `b8d46d8c3b34` 后 alembic head 从 `82b6ba3706c4` 变成 `b8d46d8c3b34`，导致原 xfail 测试里 `downgrade -1` 语义改变——原来是撤销 OrderItem 拆分，现在只撤销 unique constraint。

这是 UM870 历史反复出现的模式：跨层改动同步不彻底。

**处方**：每次改动后问"涟漪能传到多远？"然后实际去看那些文件。

有用工具：
```bash
git grep -l "\.sku\b" -- core/ modules/ tests/ scripts/
```

## 给未来的小结

1. **交付报告数字 = pytest 原始尾行**，不是解读后的版本
2. **验收测试不许重写**，发现测试本身有 bug 要在报告里显式标出
3. **测试必须真实挂钩生产代码**——删到 pass 后测试应该 fail
4. **跨层改动后 `git grep` 一下**相关名字，避免漏同步
5. **`xfail(strict=True)`** = "修好后必须移除装饰器"，不是可选保留

## 技术 Pattern（补充）

- `pytest.mark.xfail(strict=True)`：修复后必须移除装饰器，否则 strict 模式会让意外通过的测试变成 FAILED
- 迁移文件已合入 main 后不要再改，建新的 migration 做补充修改
- 测试文件若需修改，需在交付报告里显式标出并说明理由
